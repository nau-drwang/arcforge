# ArcForge Cloudflare Pages Full Stack Deployment

This project uses one GitHub repo with separated frontend/backend folders:

- Frontend source: `frontend/src/` and `frontend/public/`
- Backend source: `backend/functions/`
- Historical D1 migrations: `backend/migrations/`
- Generated frontend output: `dist/`
- Generated Cloudflare Functions mirror: `functions/`

Cloudflare Pages still deploys from the single repo.

## Cloudflare Pages settings

Build command:

```bash
npm ci && npm run pages:build
```

Build output directory:

```text
dist
```

Root directory:

```text
leave blank
```

Node.js version:

```text
22
```

Compatibility flags, if needed:

```text
nodejs_compat
```

## Bindings

Add these in Cloudflare Pages -> Settings -> Functions -> Bindings.

```text
D1 database binding
Variable name: DB
Database: arcforge-database
```

```text
R2 bucket binding
Variable name: MEDIA
Bucket: arcforge-media
```

## Environment variables

Required for the CMS login session:

```text
SESSION_SECRET = a long random string for signing admin cookies
```

Do not set `ADMIN_USERNAME` or `ADMIN_PASSWORD` for CMS v5.2. The first owner account is created inside `/admin.html`.

Redeploy after adding or changing environment variables.

## Database initialization

CMS v5.2 initializes the required D1 tables automatically when `/admin.html` opens and the API is called.

You do not need to run SQL manually in the D1 Console.

Historical migration files remain in `backend/migrations/` for documentation and developer reference.

## First-time setup

After deployment:

1. Visit `/admin.html`.
2. If no admin users exist, the installer screen appears.
3. Create the owner account.
4. Log in and use **Import seed artworks** if the artwork list is empty.

## Development rule

Edit the canonical source folders only:

- Frontend: `frontend/`
- Backend: `backend/`
- Seed/content: `data/`
- Docs: `docs/`

Do not manually edit generated `dist/` or root `functions/` unless applying an emergency deployment patch. They are regenerated by `npm run build`.


### CMS v5.2 admin setup

`SESSION_SECRET` is recommended as a Secret in Cloudflare Pages Variables & Secrets. CMS v5.2 no longer blocks setup if Cloudflare variable propagation fails; it can generate a persistent internal D1 session secret and continue. Visit `/admin.html` after deployment to create the first Owner account.


## CMS V5.2 MVP

Removed the misleading SESSION_SECRET configuration notice from the admin UI. The admin flow now relies on the backend session endpoint and goes directly to Owner setup, Login, or Dashboard.


## ArcForge CMS V5.2

CMS V5.2 is the first repairable CMS MVP. It keeps the one-repository Cloudflare Pages architecture, but fixes the prior admin setup loop. The admin backend now returns a setup state from `/api/admin/session`:

- `setup_required`: no admin user exists, so show Create Owner.
- `reset_allowed`: a previous unfinished setup created an admin user but no successful login has happened yet, so the Owner account can be safely recreated.
- `authenticated`: show Dashboard.
- otherwise: show Login.

No manual D1 migration is required. Do not set `ADMIN_USERNAME` or `ADMIN_PASSWORD`. Keep `SESSION_SECRET` as a Cloudflare Secret if it is already configured.
